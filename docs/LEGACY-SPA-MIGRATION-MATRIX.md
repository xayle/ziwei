# Legacy / SPA 迁移矩阵（2026-04-24）

## 目标

把当前仍保留的 legacy 页面与 SPA 页面做逐页对照，明确：

- 哪些页面已经有新版对应页
- 哪些能力已经迁移完成
- 哪些仍存在功能缺口
- 哪些页面暂时不能下线

---

## 总体结论

当前项目已完成“主入口统一到 SPA 优先”的第一阶段，但**legacy 页面本身仍未完成逐页迁移**。

现状更准确的描述是：

- 入口层：已收口为“唯一主站 `/static/app/cases` + 其余 SPA 子路由 + legacy 回退”
- 页面层：仍处于“核心功能已迁移，运营/工具面板未完全迁移”的混合阶段

#### `ziwei.html` 兼容跳转页方案评估（2026-04-25）

当前 `ziwei.html` 的退役形态，理论上有三种：

| 方案 | 描述 | 优点 | 风险 | 当前建议 |
|---|---|---|---|---|
| 方案 A：继续长期保留 legacy 兜底页 | 保持现状，不做自动跳转 | 风险最低 | 会长期保留双入口与维护成本 | 仅适合作为短期观察，不建议成为长期方案 |
| 方案 B：改为兼容跳转页 | 默认优先跳到 `/static/app/ziwei`，允许通过 `?legacy=1` 停留旧版 | 与 `bazi.html` / `admin.html` 策略统一；逐步观察风险；方便退役前过渡 | 仍需保留一段时间的 legacy 代码与提示条 | **推荐** |
| 方案 C：直接退役并强制重定向 | 直接让 `/static/ziwei.html` 跳到 SPA 或删除 | 入口最干净 | 若仍有隐藏运营链路，会直接阻断使用 | 当前不建议直接执行 |

#### 推荐方案：采用“兼容跳转页”两阶段策略

当前最稳妥的做法，不是继续长期保留完整 legacy，也不是立刻硬切，而是采用与 `bazi.html` / `admin.html` 相同的兼容跳转策略：

##### 阶段 1：人工回归通过后，改为兼容跳转页

目标：

- 默认用户进入 SPA `ZiweiView`
- 保留 `?legacy=1` 作为显式旧版入口
- 页面顶部增加兼容提示，明确当前页不再是主入口

最低实施项：

1. 在 `static/ziwei.html` 头部加入 SPA 可用性探测脚本
2. 默认优先跳到 `/static/app/ziwei`
3. 支持 `?legacy=1` 保持停留旧版
4. 在 README / 交付文档 / 迁移矩阵中把 `ziwei.html` 标注为 compat 页面

##### 阶段 2：观察一轮后，再决定是否彻底退役

观察重点：

- 是否还有真实用户必须依赖 `?legacy=1`
- 是否还有运营链路只在 legacy 可完成
- 是否出现 SPA 无法覆盖的边缘场景

若观察期内没有新的阻塞，可再进入最终退役讨论。

#### 当前结论（更新）

对 `ziwei.html` 而言，当前更合适的不是“立刻下线”，而是：

> **先完成人工回归，再转为 compat 跳转页观察。**

这意味着：

1. 当前阶段先不直接改 `static/ziwei.html`
2. 优先完成运营场景人工回归
3. 回归通过后，再复制 `bazi.html` / `admin.html` 的 compat 跳转模式

#### 与当前 SPA 的关系（2026-04-24 更新）

需要明确区分两类“批量”能力：

1. `static/batch.html`
   - 面向 **八字 verify / 核验**
   - 输入是 CSV 行数据
   - 核心接口是 `/api/v2/verify`
   - 输出是逐行核验结果与导出 CSV

2. `frontend/src/views/ZiweiView.vue` 中的批量面板
   - 面向 **紫微批量排盘**
   - 核心接口是 `POST /api/v1/ziwei/batch`
   - 输出是 ZIP 下载

因此它们虽然都叫“批量”，但并不是同一页面职责，不能简单视为“已有 SPA 对应页”。

#### 去向判断（新增）

`batch.html` 当前更适合被视为：

- 一个独立的专业工具页
- 而不是 `ZiweiView` 的遗留缺口

短期内不建议为了“统一入口”而强行把它塞进现有 SPA 紫微页，原因包括：

1. 业务对象不同：`batch.html` 是八字 verify，`ZiweiView` 是紫微排盘
2. 输入输出不同：前者是逐行核验表格，后者是 ZIP 打包下载
3. 使用人群不同：前者更偏测试/运营工具，后者更偏业务命盘生成

更合理的两个长期选项是：

- 方案 A：长期保留为独立工具页
- 方案 B：未来迁入独立的 SPA 工具页（如 `/static/app/tools/batch-verify`），但不并入 `ZiweiView`

#### 判断

这是当前最明确的“仍应独立保留”的页面之一。

在没有独立 SPA 工具页前，不应下线。

#### 当前建议（更新）

1. 短期继续保留 `static/batch.html`
2. 文档中明确它是“独立工具页”，不要误记为 `ZiweiView` 缺口
3. 若后续要迁移，应新建独立 SPA 工具页，而不是塞回紫微主盘页
