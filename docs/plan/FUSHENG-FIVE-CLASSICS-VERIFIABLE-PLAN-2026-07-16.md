## 一、背景与问题

### 1.1 产品要什么

讲解层应对齐可核验的子平典籍依据：有书证才硬标「典籍依据」，无据不妄断，禁套话迎合；引擎事实与典籍分层可读。

### 1.2 现状差距（2026-07-16 盘点）

| 书 | 库内条数（标题含书名） | verified | 诊断 |
|----|------------------------|----------|------|
| 《滴天髓》 | ~179 | ~63 | 较厚；需按类目补齐并加核 |
| 《子平真诠》 | ~88 | ~23 | 用神/十神/格局加核 |
| 《穷通宝鉴》 | **~4** | **~1** | **结构近空 · 战役主攻** |
| 《三命通会》 | ~49 | ~3 | 调候/神煞/十神加核 |
| 《渊海子平》 | ~22 | ~1 | 神煞/岁运/十神加核 |

另有问题：

- `classical_narrative` 等 **软句式** 易被读成典籍；  
- cite 已要求 verified，但五书结构未齐时，「强典籍」叙事名不副实；  
- 他书（千里、神峰、易经）混在语料中，不宜一句清掉，也不宜冲淡终点。

### 1.3 为什么是「结构+verified」而不是「全文先囤」

全文入库不等于可引用；未核对的长文会诱导产品说谎。本计划终点是：**类目有代表条，且关键条已核验**——不限周期，但每一步都须可产品化。

## 二、目标与非目标

### 2.1 总目标

1. 五书在产品类目矩阵上 **无空白**（每应有格 ≥1 条 verified）。  
2. 关键条目 **大量 verified**，达到 §4 数字门禁。  
3. 工程侧消灭「软句假典籍」；缺证处诚实空态。  
4. 达标后方可对外强化「典籍可核对」叙事，并考虑 cite 白名单升格。

### 2.2 非目标

| 不做 | 原因 |
|------|------|
| 五书扫描件全文无条目化入库即算完成 | 无法 cite、无法抽检 |
| LLM 编造句标 verified | 违反内容宪法 |
| 删光非五书语料 | 误伤对照与神煞参考；终点不含「只留五书档案」 |
| 本计划内做完 GTM/河洛/奇门 | 另战役 |
| 用赶工数字牺牲 spotcheck | 周期不限，质量优先 |

### 5.5 单条语料标准工序（强制）

```text
选题（类目矩阵缺口） 
  → 底本摘录（可复述但须可回溯）
  → 写入 classics.json（id/title/passage/tags/…）
  → spotcheck 登记
  → verification_status=verified
  → （如需）classic_refs / explain 命中测
  → E-03 报表更新
```

**禁止：** 未 spotcheck 直接 verified；用模型生成句冒充原文。
