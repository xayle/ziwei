# 项目计划 - 2026-06-15

## 目标

在不打断现有功能交付的前提下，整理当前文档口径，确认真实功能边界，补齐质量门禁，并为后续的结构优化和版本演进建立一份可执行计划。

## 当前基线

- 后端已挂载多个业务域路由，核心能力覆盖八字、紫微、合婚、成员、事件、场景、报告、LLM、风水、姓名、西占、塔罗、六爻、析取/相似盘等模块。
- 前端已存在 SPA 路由，覆盖首页、登录、八字、紫微、紫微批量、紫微案例页、报告、风水、择日、西方占星、数字学、合婚、多人合盘、术语词库、人类设计、塔罗、成员管理、场景推演、AI 草稿和六爻等页面。
- 文档体系已分层为 `design`、`reports`、`guides`、`samples`、`archive`，但仍存在少量旧链接和历史口径混用。
- 质量基线文档显示：测试、构建、静态检查并非同一时点同步更新，必须以当前代码和当前测试结果为准。

## 计划原则

1. 先统一事实来源，再做结构整理。
2. 所有计划项都要能映射到代码、路由、测试或文档。
3. 旧文档保留，但必须明确标记历史属性。
4. 新增能力优先进入 SPA 和当前 API 体系，不再扩散旧入口。
5. 每一步改动都要有回归验证。

## 执行阶段

### Phase 1: 文档整理

目标:

- 建立统一文档入口。
- 区分现状文档、设计文档、报告、指南、样例、归档。
- 清理 README 中失效的文档路径。

交付物:

- `docs/README.md`
- 根目录 `README.md` 的文档导航更新
- 旧文档引用映射表

### Phase 2: 功能盘点

目标:

- 以代码和路由为准，重建当前功能地图。
- 按业务域列出“已完成、可用、待验证、历史遗留”。

重点模块:

- 八字
- 紫微
- 成员与事件
- 场景与案例
- 报告书
- 合婚与多人关系
- 姓名学
- 西方占星
- 择日
- 风水
- 塔罗
- 六爻
- LLM 与相似盘

交付物:

- 当前功能清单
- 功能状态表
- 路由与页面映射表

### Phase 3: 质量基线复核

目标:

- 重新核对测试、构建、类型检查、静态分析、关键页面可用性。
- 把历史基线和当前基线分开。

交付物:

- 当前测试结果
- 当前构建结果
- 当前静态分析结果
- 当前风险清单

### Phase 4: 结构优化准备

目标:

- 识别超大页面、过载入口和重复逻辑。
- 明确哪些模块适合继续拆分，哪些先保持稳定。

优先对象:

- 后端入口装配层
- 前端档案与分析入口
- 紫微页面族
- 八字分析页面族
- 报告书编排层

交付物:

- 技术债清单
- 拆分优先级
- 低风险重构建议

### Phase 5: 门禁与回归

目标:

- 把关键路径纳入固定回归门槛。
- 让文档、测试、构建保持同步。

建议门禁:

- 后端测试
- 前端单测
- 前端类型检查
- 前端构建
- 关键接口契约检查

交付物:

- 最小质量门禁定义
- CI 执行建议

### Phase 6: 版本推进

目标:

- 将整理工作转化为后续版本迭代。

版本方向:

- v8.x: 文档收敛、口径统一、质量门禁稳定
- v9.0: 结构拆分、入口收敛、页面模块化
- v9.x+: 持续优化、性能治理、跨模块编排优化

## 近期优先级

1. 建立当前文档索引和计划入口。
2. 修正 README 中失效的文档链接。
3. 输出当前功能列表和模块状态表。
4. 复核测试与构建基线。
5. 给档案中心、紫微、八字、报告书做拆分与治理排期。

## 完成标准

- 文档入口清晰，旧链接不再误导。
- 当前功能地图可直接用于沟通和排期。
- 质量基线有明确“当前值”。
- 下一轮开发可以直接按计划拆分执行。
