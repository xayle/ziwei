### 决策2: 隐私模型(Privacy Model)的最终定义

**问题**: 方案说"不存储"，但URL分享链接会在多个位置留下生日信息

**现在的问题**:
```
用户说：我不想别人知道我的生日
我们说：我们不存。

但实际上：
- 浏览器历史记录: 有生日 ✗
- ISP日志: 有生日 ✗
- CDN日志: 有生日 ✗
- 搜索引擎索引: 可能有生日 ✗
- 分享给别人: 对方知道生日 ✗

这是对用户的欺骗。
```

**选项A**: **完全隐私模式（生日不过网络）**
```
做法：
1. 使用本地计算（WebAssembly）
2. 所有sxtwl/cnlunar逻辑编译成WASM
3. 浏览器完全本地计算，不
4. 生日永远不上传服务器

缺点：需要WASM，开发工作量大
优点：真的隐私
```

**选项B**: **诚实模式（承认风险）**
```
做法：
1. 改隐私声明为"真实"
2. 明确说"链接会包含生日，不要分享给陌生人"
3. 提供'仅限此设备'计算（使用localStorage）
4. 不支持分享链接

缺点：不能分享
优点：简单，诚实
```

**选项C**: **当前做法（保持现状）**
```
缺点：
- 隐私声明与实际不符
- 欧盟GDPR风险
- 用户如果起诉，我们无法辩解
```

**当前状态**: ❌ **未决策**（方案v5.2中有矛盾）

**建议**:
- [ ] 如果是娱乐项目 → 用**选项B**（诚实+简单）
- [ ] 如果要上线商业化 → 用**选项A**（WASM隐私）
- [ ] 绝对不要用**选项C**（风险太大）

**风险**: 🔴 **高** - 法律+隐私问题

### 决策5: 主引擎 + 备份引擎的处理方式

**当前设计**:
```
主引擎: sxtwl（比较准）
备份引擎: cnlunar（作为参考）

如果两个结果不同：
- L2或L1级别（不够可信）
- 两个四柱都展示给用户

问题：
1. 这样对用户很混乱（为什么有两个答案）
2. 推荐规则没有明确说"用哪个引擎的十神"
3. 没有办法自动判断"哪个引擎对"
```

**选项A**: **完全信任主引擎，备份作为参考**
```
- 十神、推荐全部基于sxtwl
- cnlunar只在有差异时"显示"给用户参考
- L级别基于"主引擎是否有boundary_risk"

代码简单，逻辑清晰
```

**选项B**: **加权平均或投票（如果两库都能计算）**
```
- 如果sxtwl和cnlunar结果一致 → L0（高可信）
- 如果结果不一致 → L1（低可信，展示两个）

这需要对两个库的所有计算都实现
```

**当前状态**: ⚠️ **部分确定**（方案说用sxtwl为主，但代码未完全遵循）

**建议**:
- [ ] 确认选项A或B
- [ ] 如果选A，改计算逻辑（cnlunar完全作为备份，不影响十神）
- [ ] 如果选B，确保cnlunar的所有计算都实现

**风险**: 🟠 **中** - 混乱会影响用户体验

### 可能遗漏或需要深化的地带⚠️

1. **并发性和竞态条件** (Concurrency)
   - 考虑过吗：两个用户同时请求会出现什么问题
   - 当前SQLite有并发锁吗？一个请求多长时间会导致另一个超时
   - sxtwl库的线程安全性验证了吗

2. **错误恢复策略** (Error Recovery)
   - 如果sxtwl库某个版本有bug，怎么快速回滚
   - 如果某个用户的请求导致内存溢出，会影响其他用户吗
   - 有graceful degradation吗（比如实在算不出来，能否给个默认值或友好提示）

3. **性能基准和优化目标**
   - API响应时间目标是多少（毫秒）？
   - 用户量从1增加到1000时，系统能否支撑
   - 有缓存策略吗（比如同一个八字多次查询）

4. **无损回滚计划**
   - 当前代码上线后，如果发现大bug，能否秒速回滚到前一版本
   - 数据库schema变更后如果出问题，能否回滚并恢复数据

5. **版本管理和兼容性**
   - requirements.txt是否固定了所有依赖的版本号
   - 如果六个月后，Python 3.12不支持sxtwl 1.1，怎么办
   - API版本策略（/api/v1/, /api/v2/)的演进计划

6. **本地化和国际化** (I18N)
   - 如果未来要支持英文，前端有没有考虑
   - 十神、神煞的英文翻译准备好了吗
   - 日期格式（农历/阳历/一个混合）的展示逻辑清楚吗

7. **生成的请求/响应的示例数据**
   - test_payload.json和verify.html中的样例数据和谐吗
   - 有没有"黑名人"的生日进入代码（引起争议）

8. **数据导出/分享的格式**
   - 目前计划支持什么格式（JSON/PDF/图片）
   - 生成的文件有没有考虑长期保存的可读性

9. **第三方库的更新周期**
   - sxtwl最后更新是何时
   - cnlunar最后更新是何时
   - FastAPI/Pydantic的小版本更新是否会破坏兼容性

10. **监控和告警的实施计划**
    - 虽然文档提到"需要监控"，但具体怎么开始
    - 有ELK stack/Prometheus吗，还是用简单的logging
