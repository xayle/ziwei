# 个人档案型命理产品前端设计稿

版本：v1.1
日期：2026-06-19
状态：可执行设计稿

## 1. 产品目标

把当前前端收敛成一个稳定的 `个人档案型命理交付工具`。

核心不是聊天，不是社区，不是娱乐页，而是：

1. 基础信息输入。
2. 生成个人数据卡。
3. 输出八字、紫微、姓名结果。
4. 形成正式 PDF 档案。
5. 支持人工注释讲解。

最终用户拿到的内容，应当像一份可交付、可讲解、可留档的正式文件，而不是一个功能堆叠的命理控制台。

## 2. 产品边界

这个产品应当被理解为 `档案交付工具`，而不是：

- 聊天机器人
- 玄学社区
- 娱乐测试页
- 门户式命理导航站
- 多入口堆叠

不做的事：

- 不重写底层命理计算逻辑
- 不迁移技术栈
- 不扩展成社交产品
- 不在首页直接暴露完整分析结果

## 3. 设计原则

1. 首页只保留输入和预览。
2. 档案页作为标准化资料编辑中心。
3. 报告页作为正式交付件。
4. 所有页面统一为一套正式、克制、档案感视觉语言。
5. 所有输出遵循 `先摘要、再结构、后解释`。
6. 长文本拆短，复杂内容分层，不在一个区块里塞完。
7. 报告页预览与 PDF 章节结构保持同构。
8. 所有导出要保留人工注释的可讲解空间。

## 4. 当前前端状态

当前仓库已经具备以下页面骨架：

- `HomeView`
- `ProfileView`
- `ReportView`

并且相关组件已经拆分到：

- `frontend/src/components/home/`
- `frontend/src/components/profile/`
- `frontend/src/components/report/`

这意味着当前工作重点不是再造技术栈，而是统一信息架构、页面职责、视觉语法和导出结构。

## 5. 页面职责

### 5.1 首页

首页只做三件事：

1. 填基础信息。
2. 预览个人数据卡。
3. 进入报告主链路。

首页不承载：

- 八字完整分析
- 紫微完整分析
- 姓名完整分析
- 复杂入口堆叠
- 多余的次级工具并列展示

### 5.2 档案页

档案页是基础资料编辑中心，职责是把输入标准化。

它要解决的问题是：

- 基础信息是否完整
- 时间信息是否可信
- 地点信息是否标准化
- 现居地和历史事件是否可复用

档案页不承担正式报告内容。

### 5.3 报告页

报告页是正式交付件，职责是把计算结果整理成可阅读的档案。

它要呈现：

- 封面
- 目录
- 基础档案
- 时间轴
- 八字总览
- 紫微总览
- 姓名分析
- 综合总结
- 人工注释区
- 附录与免责声明

报告页不能再看起来像控制台。

## 6. 首页设计

### 6.1 页面目标

- 让用户快速开始填写
- 让用户能即时看到自己的档案状态
- 让用户一键进入报告生成流程

### 6.2 页面结构

- 顶部 Hero
- 左侧输入区
- 右侧个人数据卡预览
- 底部主操作区

### 6.3 左侧输入区

按四组组织：

1. 基础信息
2. 时间设置
3. 现居地
4. 历史事件 / 当前关注

### 6.4 右侧预览区

按三层展示：

1. 基础档案摘要
2. 时间与地点摘要
3. 现实标签与状态摘要

### 6.5 底部按钮

只保留一个主按钮：

- `生成档案并进入报告`

次级按钮尽量少，避免首页入口混乱。

### 6.6 首页保留内容

- 姓名
- 性别
- 出生年月日
- 出生时间
- 出生地
- 时区
- 经度
- 历法
- 闰月
- 时辰精度
- 未知时辰兜底
- 真太阳时
- 现居地
- 当前关注重点
- 历史事件
- 档案完整度
- 时间可信度

### 6.7 首页砍掉内容

- 八字完整分析入口
- 紫微完整分析入口
- 姓名完整分析入口
- 工具台入口
- 娱乐化文案
- 多余跳转按钮

### 6.8 首页完成标准

- 用户一眼知道该怎么填
- 只有一个主路径
- 首页像档案入口，不像控制台

## 7. 档案页设计

### 7.1 页面目标

- 把资料标准化
- 统一出生信息、时间信息、地点信息
- 给后续结果提供稳定基础

### 7.2 页面结构

- 顶部说明
- 中间编辑区
- 右侧摘要卡
- 底部保存 / 重置 / 前往报告

### 7.3 编辑区分组

1. 基础信息
2. 时间与地点
3. 现居地
4. 分析偏好
5. 历史关键事件

### 7.4 右侧摘要

固定显示：

- 档案完整度
- 时间可信度
- 当前关注重点
- 关键标签
- 当前补全状态

### 7.5 档案页保留内容

- 姓名、性别、出生时间、出生地、时区、经纬度
- 公历 / 农历切换
- 闰月
- 时辰精度
- 未知时辰兜底
- 真太阳时
- 现居地
- 当前关注重点
- 历史关键事件

### 7.6 档案页新增内容

- 档案完整度提示
- 时间可信度提示
- 当前补全状态
- 统一的档案摘要区

### 7.7 档案页砍掉内容

- 正式报告正文
- 长篇命理解释
- 结果页式排版
- 控制台式入口

### 7.8 档案页完成标准

- 它是唯一的标准化编辑中心
- 所有字段服务于后续报告生成
- 只输出稳定、可复用的档案契约

## 8. 报告页设计

### 8.1 页面目标

- 让它像正式交付件
- 能直接讲解、打印、留档
- 不再像控制台或多入口导航站

### 8.2 页面结构

- 左侧章节索引
- 中间正文
- 右侧摘要 / 预警 / 注释

### 8.3 章节顺序

1. 封面
2. 目录
3. 个人基础档案
4. 时间轴
5. 八字总览
6. 紫微总览
7. 姓名分析
8. 综合总结
9. 人工注释区
10. 附录与免责声明

### 8.4 章节原则

- 每章先给摘要
- 再给结构信息
- 最后给解释
- 不把原始计算过程直接铺满正文

### 8.5 报告页保留内容

- 封面
- 目录
- 个人基础档案
- 时间轴
- 八字总览
- 紫微总览
- 姓名分析
- 综合总结
- 人工注释区
- 附录与免责声明

### 8.6 报告页新增内容

- 主动预警区
- 章节锚点导航
- 结构摘要卡
- 与 PDF 同构的章节语法

### 8.7 报告页砍掉内容

- 表单编辑能力
- 控制面板式布局
- 复杂调参入口
- 和报告无关的工具内容

### 8.8 报告页完成标准

- 作为正式交付件
- 所有章节遵循 `先摘要、再结构、后解释`
- 网页预览与 PDF 保持同构

## 10. 视觉系统

### 10.1 颜色

- 暖米白

- 深墨色
- 低饱和青绿色

### 10.2 语义

- 金色：主状态

### 10.3 风格关键词

- 档案

- 纸张

### 10.4 禁止方向

- 紫色霓虹
- 玄学闪光
- 娱乐插画
- 门户站密集排版
- 工具台式信息堆叠

## 11. 内容语法

统一所有页面和章节的表达方式：

1. 先结论
2. 再结构
3. 后解释

这意味着：

- 首页的预览卡只放摘要
- 档案页的右侧卡只放状态和关键信息
- 报告页的正文才进入完整分析

## 13. 当前实现约束

1. 保留 Vue + Vite + Router 技术栈。
2. 不迁移框架。
3. 不重写底层命理计算逻辑。
4. 优先改页面结构和视觉语法。
5. 报告页和 PDF 预览要尽量保持同构。

## 14. 前端实施顺序

### 阶段 1

- 固定首页布局
- 固定档案页布局
- 固定报告页骨架

### 阶段 2

- 统一首页 / 档案页 / 报告页视觉语法
- 收敛按钮和入口
- 减少控制台感

### 阶段 3

- 统一八字 / 紫微 / 姓名的摘要结构
- 接入时间轴
- 接入主动预警区

### 阶段 4

- 对齐 PDF 导出和页面预览
- 校验分页和章节顺序
- 补人工注释区

## 15. 交互状态

### 15.1 加载态

- 页面骨架屏
- 章节加载中
- 导出按钮置灰

### 15.2 空状态

- 首页：提示补全基础档案
- 档案页：提示补全基础信息
- 报告页：提示先选择或生成案例

### 15.3 错误态

- 报告加载失败时保留重试按钮
- 导出失败时给出明确错误提示
- 表单字段异常时在档案页局部提示

## 16. 响应式与打印

### 16.1 桌面端

- 首页：左右双栏
- 档案页：编辑区 + 摘要区
- 报告页：三栏结构

### 16.2 移动端

- 首页和档案页纵向堆叠
- 报告页改为单栏阅读
- 右侧批注区折叠为底部面板或抽屉

### 16.3 打印 / PDF

- 章节顺序固定
- 页面分页明确
- 封面独立
- 目录单独成页
- 时间轴和预警区不破版

## 17. 验收标准

1. 用户进入首页后，只看到输入与预览，不会被复杂入口干扰。
2. 档案页成为唯一的标准化资料编辑中心。
3. 报告页呈现正式交付件气质。
4. 八字、紫微、姓名都遵循统一摘要结构。
5. 时间轴和预警区在报告中可见。
6. PDF 和网页预览结构一致。
7. 旧功能不回退，主流程稳定可用。

- `HomeView`
