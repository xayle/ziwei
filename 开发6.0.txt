前置决策（N1.01 开始前必须确认，不确认就没有代码依据）：

  [D1] 化气格判定标准（修订版，以本决策块为准，N1.02 正文为实现细节）
    核心前提：日主天干必须参与五合（《子平真诠》明确以日主合化为基准）
    判定方式：day_stem 与某一柱（年/月/时）天干成五合 + 月支归属五行 = 化出五行
    ⚠️ 废除旧表述"四柱任意两柱天干成合（不限相邻）"——该表述会造成 5/6 概率误判
       6 种组合对中只有 3 对含日主（月日/年日/时日），其余 3 对不构成化气格
    放弃《滴天髓》更严格的无克制条件
    边界争议通过 confidence 分值表达（化气格最高给 0.85，不给 1.0）

  [D2] 破格不做
    geju.py 注释已明确"破格在 M3 补充"，本轮不实现破格逻辑
    confidence < 0.5 在语义上代替"疑似破格"，UI 显示待定标签

N1.01 compute_geju() 新增 confidence 字段
  - 文件：services/bazi_engine/geju.py
  - 修改 compute_geju() 返回 dict 新增键：confidence: float（0.0-1.0）
  - 计算规则：
      正格透干判定  → confidence = 0.85
      正格无透干靠藏干 → confidence = 0.65
      建禄格（月令日主临官）→ confidence = 0.80（月令司令，成格清晰）
      羊刃格（月令日主帝旺）→ confidence = 0.80（刃旺成格，吉凶争议大但格局成立）
      ⚠️【命理定义修正】建禄格月令条件是日主"临官"位（禄刃表），非帝旺位
        帝旺位 = 刃位 = 羊刃格；两格月令位置不同，干支查表结果不同
        日主甲：临官=寅（建禄），帝旺=卯（羊刃）
        日主丙：临官=巳（建禄），帝旺=午（羊刃）
        其余日主以十二长生表类推，禁止将两格月令条件互换
      ⚠️ 建禄/羊刃格用神取法与正格完全不同，yongshen 计算必须走专用分支：
        正格：月令之神为用神根本（月令生旺日主的五行即用）
        建禄/羊刃：月令本身即日主旺气，无法再以月令为用
        → 改为看四柱透干，取官/杀/财/食/印中最强透干为用（《子平真诠》建禄格篇原则）
        实现要求：compute_geju 返回的 geju_name 含"建禄"或"羊刃"时，
          yongshen 计算器检查此标志，走非月令用神分支，否则用神全算错
      从旺格（某五行 ≥70% 且该五行 == 日主五行）→ confidence = min(0.5 + ratio*0.5, 0.95)
      专旺格（某五行 ≥70% 且该五行 != 日主五行）→ confidence = min(0.4 + ratio*0.4, 0.85)
      ⚠️ 两者用神取法截然相反，禁止用同一公式：
         从旺格取比劫/印为用；专旺格绝对禁用比劫（顺则吉，逆则凶）
         confidence 值差异是后续用神推算区分两种格局的唯一依据
      化气格（D1 标准满足）→ confidence = 0.70
        + 命局中无任何五行克化出五行 → 再加 0.10
        （"无克制"定义：四柱天干+地支主气中，无任何元素的五行克化出五行
          例：化土格，命局无木干/木支主气；化火格，命局无水干/水支主气
          禁止将月支条件重复用于此判断，D1 已包含月支约束）
      普通格 → confidence = 0.40（触发 UI 待定标签）
  - 不新建函数，在现有 compute_geju 函数内追加计算逻辑

N1.02 化气格判定（5 种天干五合）
  - 在 geju.py 新增 _check_huaqi(stems: list[str], month_branch: str) 内部函数
  - ⚠️ stems 参数必须传入 [year_stem, month_stem, day_stem, hour_stem] 全部四干
    现有 compute_geju 中的 candidate_stems 刻意排除了 day_stem（十神计算不自克）
    但化气格取的是天干五合，日主天干参与五合恰是最典型情况（甲日己月=化土格）
    禁止复用 candidate_stems 变量，必须单独构造含 day_stem 的完整四干列表
  - 五合规则：甲己化土/乙庚化金/丙辛化水/丁壬化木/戊癸化火
  - ⚠️【命理核心约束，修订 D1】化气格成立必须由日主天干参与五合
    命理正典（《子平真诠》）明确：化气格以日主合化为基本前提
    若年干甲与月干己相合但日主为丙火 → 丙火不化土，此命局不是化土格
    有效组合（日主天干必须是合化的两干之一）：
      月日合（month_stem+day_stem）/ 年日合（year_stem+day_stem）/ 时日合（hour_stem+day_stem）
    无效组合：年月合/年时合/月时合（日主未参与，4柱6对组合中5对均无效）
    禁止写"四柱任意两柱成合"——会产生 5/6 概率的误判
  - 条件（D1 修订）：day_stem 与某一柱天干成合 + 月支归属五行 = 化出五行
  - ⚠️ 月支主气五行判断实现规则（4个土支必须统一处理，否则漏判）：
      取 STEM_ELEMENT[BRANCH_HIDDEN_STEMS[month_branch][0][0]][0] 得五行英文名
      辰/丑/未/戌主气分别为戊/己/己/戊，五行均为 "earth"，化土格在此四支均成立
      判断时以五行类别（"earth"/"metal"/"water"/"wood"/"fire"）相等为准
      禁止写 main_stem == "戊" 或 main_stem == "己" 的天干精确匹配（会漏掉丑未）
  - 返回：{"is_huaqi": bool, "huaqi_element": str, "confidence": float}
  - 五种化气格名称规范（禁止自由发挥，全程精确使用以下名称）：

N3.05 权限缓存
  - 🚨【安全分离】权限缓存必须与引擎 QueryCache 物理隔离，禁止共用同一单例：
      风险1：引擎缓存清空时同步清除权限缓存 → 所有用户权限瞬时失效（拒绝服务）
      风险2：引擎缓存 key 命名冲突可污染权限查询结果 → 潜在越权访问
  - 实现：在 app/dependencies/permissions.py 创建独立 _permission_cache 单例：
      class PermissionCache:
          def __init__(self, ttl: int = 300):
              self._cache: dict[str, tuple[Any, float]] = {}
              self._ttl = ttl
          def get(self, key: str): ...    # 检查 TTL，过期返回 None
          def set(self, key: str, value): ...
          def invalidate(self, key: str): self._cache.pop(key, None)
      _permission_cache = PermissionCache(ttl=300)  # 仅此文件可见，禁止跨模块复用
  - cache key：f"perm:{user_id}:{permission_type}"（TTL = 5min）
  - N3.02 approve/reject/revoke 时主动调用 _permission_cache.invalidate(key) 清除缓存
  - 不阻塞引擎计算（满足 R36）

N3.07 RBAC 流程测试
  - 文件：tests/test_rbac_workflow.py（新建）
  - 主干流程：
      申请→批准→权限生效
      申请→拒绝→权限不生效
      普通用户调用 approve → 403
  - 边界场景（必须覆盖，否则状态机存在安全漏洞）：
      自我审批：requested_by == 当前审批用户 → 返回 403（禁止自批）
      重复审批：已 approved 再次调 approve → 返回 409（幂等会掩盖重复操作，409更安全）
      非法状态转换：approved 后调 reject → 返回 409（已批准权限须走专门 revoke 流程撤销）
      ⚠️ 上述决策为强制规范，实现者不得自行调整；测试断言必须与此一致，不接受"两种皆可"
      并发审批：两管理员同时 approve 同一申请 → 不得重复记权
        ⚠️ 必须在 N3.00 migration 时确定具体方案，不可写"乐观锁或唯一约束"留给实现者选
          强制方案：UPDATE 时加 WHERE status='pending' 乐观状态检查
            UPDATE delegations SET status='approved', approved_by=:uid, approved_at=now()
            WHERE id=:id AND status='pending'
            若 rowcount==0 → 返回 409（已被其他管理员处理，本次 approve 无效）
          PostgreSQL 路径：追加 with_for_update(skip_locked=True) 获取行锁防止竞争
          SQLite 路径：依赖 WAL 模式 + rowcount 检查（不支持 SELECT FOR UPDATE）
          N3.00 migration 同步确认 delegations.id 主键索引存在（行锁依赖索引）

N3.09 JWT / Token 鉴权策略定义（执行时机：N3.02 编码前确认）
  - ⚠️ JWT 配置是整个鉴权体系的基础，本任务须在 N3.02 执行前明确，否则 token 过期行为不可测
  - 在 app/config.py 明确以下参数（禁止硬编码，必须从环境变量读取）：
      ACCESS_TOKEN_EXPIRE_MINUTES：访问令牌有效期（推荐 30 分钟）
      REFRESH_TOKEN_EXPIRE_DAYS：刷新令牌有效期（推荐 7 天）
      JWT_SECRET_KEY：必须从环境变量读取，.env.example 注明须随机生成（≥32 bytes）
  - 刷新流程（防止用户每 30 分钟强制重新登录）：
      前端检测 AccessToken 剩余 < 5 分钟 → 静默调用 POST /api/v1/auth/refresh
      服务端验证 refresh_token（存于 HttpOnly Cookie 或独立 localStorage key）
      通过 → 返回新 AccessToken（RefreshToken 不更换，减少频繁 DB 写操作）
      RefreshToken 过期/无效 → 前端跳转登录页
  - 登出令牌失效（登出后 JWT 在有效期内仍可使用的安全缺陷）：
      方案A（推荐，需内存存储）：JWT 黑名单
        每个 JWT payload 写入 jti（uuid4），登出时加入黑名单，TTL = AccessToken 剩余有效期
        中间件每次请求检查 black_list:{jti}；命中则返回 401
      方案B（无外部依赖，可接受）：保守模式
        AccessToken 有效期缩短到 15 分钟；登出只清除客户端 token
        15 分钟内旧 token 仍可用（权衡：简单实现，15min 内安全窗口可接受）
      在 .env.example 和 README 注明所选方案及原因
  - 验收：携带已登出 token → 401（方案A）或等待过期（方案B）；携带有效 token → 200

N5.03 批量计算 API（依赖 N6.01 路由注册）
  - 端点：POST /api/v2/batch/verify
  - 请求：{"items": List[VerifyRequest]}，validator 限制 len ≤ 50（满足 R39）
  - ⚠️ 50项×5s=250s 远超整体60s，两个超时不可独立存在，必须统一设计：
    整体60s 为硬上限（通过 start_time + deadline 控制循环退出）
    per-item 5s 为每个 future 的软超时（future.result(timeout=5)）
    正确实现（Windows兼容，禁止使用 signal.SIGALRM）：
      import time
      from concurrent.futures import ThreadPoolExecutor, as_completed, TimeoutError
      start = time.monotonic()
      deadline = 60.0  # 整体硬上限60s
      with ThreadPoolExecutor(max_workers=int(os.getenv("BATCH_WORKERS", "4"))) as executor:
          # 💡 BATCH_WORKERS 从环境变量读取，可按服务器 CPU 核数调优（默认4）
          futures = {executor.submit(calc_single, item): i
                     for i, item in enumerate(items)}
          try:
              for future in as_completed(futures, timeout=max(0, deadline-(time.monotonic()-start))):
                  i = futures[future]
                  try:
                      results[i] = future.result(timeout=5)  # per-item软超时
                  except (TimeoutError, Exception) as e:
                      failed.append({"index": i, "error": str(e)})
          except TimeoutError:
              # ⚠️ 必须有外层捕获：整体60s到达时 as_completed 自身抛出 TimeoutError
              # 若不捕获 → 接口直接 500，已完成结果全部丢失
              for f, i in futures.items():
                  if i not in results:
                      failed.append({"index": i, "error": "overall timeout (60s)"})
  - ⚠️ 超时降级必须明确：
      超时触发时返回已完成的部分结果，未完成的计入 failed 列表
      格式：{"results": [...已完成], "failed": [{"index":n,"error":"timeout"}]}
      禁止整批超时后返回 502/504（用户 50 条数据全丢，无法重试哪几条）
  - 响应：{"results": List[VerifyResponse], "failed": List[{"index":int,"error":str}]}
  - ⚠️ results 字段必须为有序 List，不可返回 dict：
      内部用 dict[int, VerifyResponse] 按 index 暂存，最终输出前重建有序列表：
        ordered = [results[i] for i in sorted(results.keys())]
      failed 的 slot 不填充 None（直接跳过），index 字段指明原始位置
      客户端通过 failed[].index 知道哪几条失败，通过 results 列表得到其余成功项
      总数量关系：len(results) + len(failed) == len(request.items)（必须在测试中断言）
  - ⚠️ 每用户速率限制（防恶意并发攻击，批量接口风险高于单次接口）：
      💡【实现发现】services/rate_limit.py 已存在 slowapi Limiter，无需重新安装
         当前 _rate_limit_key() 按 IP 限速（get_remote_address），需改造为 user-based：
           改造方案（不新增依赖）：
             def _rate_limit_key(request):
                 if os.environ.get("AUTH_BYPASS","").lower() == "true":
                     return str(uuid.uuid4())  # 测试模式：每请求唯一key，不触发限制
                 token = request.headers.get("Authorization","").replace("Bearer ","")
                 user_id = decode_token_user_id(token)    # 从 JWT 解出 user_id
                 if user_id:
                     return f"user:{user_id}"             # 已登录：按用户限速
                 return get_remote_address(request)       # 未登录：降级为 IP
           @limiter.limit("10/minute") 装饰 /api/v2/batch/verify 端点
      ⚠️ 速率限制必须 per-user（按 user_id），不可 per-IP（代理后多用户共享同一 IP）
      ⚠️ 触发 429 时先拒绝，不占用 BATCH_WORKERS 线程池（验证配额后再分配资源）
      响应格式：HTTP 429 + Retry-After: 60 响应头（告知客户端等待时间）

N6.02 v2 Schema 定义
  - 文件：app/schemas/v2/verify.py（新建）
  - VerifyRequestV2：继承 VerifyRequest，追加 output_format: Literal["full","minimal"] = "full"
  - ⚠️ output_format="minimal" 必须有完整实现，否则是无效假功能（API 文档声明但无效）：
      minimal 保留字段：geju / yongshen / dayun_current / wuxing_scores / score
      minimal 省略字段：palace / shensha / classic_ref_list / monthly_fortune / lifestyle / milestone
      路由 handler 根据 output_format 跳过对应引擎调用（同时节省计算时间）
      禁止 full 和 minimal 返回完全相同的响应体（若无法区分则删除此参数）
  - ⚠️ 类型冲突：VerifyResponseV2.data:VerifyResponse 含必填字段，minimal省略后 Schema 验证必然失败
    强制方案（选A，禁止实现者随机选择）：
      方案A：定义 VerifyResponseMinimal（只含5个保留字段，均为 Optional），
             VerifyResponseV2.data 类型为 Union[VerifyResponse, VerifyResponseMinimal]
      ⚠️ Pydantic V2 无 discriminator 的 Union 按顺序匹配：VerifyResponse 是 Minimal 超集
         第一个永远匹配成功 → full/minimal 均返回 VerifyResponse → output_format 完全无效！
         强制实现 discriminator：
           ⚠️【v1 向后兼容】discriminator 字段绝对不可加在 VerifyResponse（v1响应体）上
              VerifyResponse 是 /api/v1/verify 的响应体，加新字段会破坏所有现有v1客户端
              正确做法：在 v2 专属包装层实现，v1 VerifyResponse 保持不变
           正确实现：
             class VerifyResponseFull(VerifyResponse):   # v2内部子类，继承v1但加discriminator
                 response_type: Literal["full"] = "full"
             class VerifyResponseMinimal(BaseModel):
                 response_type: Literal["minimal"] = "minimal"
                 geju: Optional[...]; yongshen: Optional[...]; ...  # 仅保留字段
             VerifyResponseV2.data 类型为 Annotated[
                 Union[VerifyResponseFull, VerifyResponseMinimal],
                 Field(discriminator="response_type")]
           v1 的 VerifyResponse 类无任何改动，/api/v1/verify 响应体结构不变
      （方案B：data:dict，失去类型安全，备选仅当 VerifyResponse 字段变化频繁时）
  - VerifyResponseV2：采用组合模式而非继承 VerifyResponse
    ⚠️ 不继承的原因：v1 VerifyResponse 改动会污染 v2 Schema，无法独立演进
    💡【继承 vs 组合 澄清，两者并存不冲突，实现者看到两者同时存在时不可随意删一个】：
      VerifyResponseFull(VerifyResponse) → 内部继承（局部使用）：
        仅为给 v2 内部添加 response_type 判别字段（Pydantic discriminator 要求）
        此类对 v1 客户端完全不可见，不影响 /api/v1/verify 响应体结构
        → 适度继承，仅为解决技术问题，不代表"v2=v1的超集"
      VerifyResponseV2(BaseModel) → 外部组合（外层信封）：
        v2 外层 meta + data 结构，不继承任何 v1 类，保证 v2 schema 独立演进
        v1 VerifyResponse 字段变化不传播到 v2 外层格式
      总结：VerifyResponseFull 在 v2 内部局部继承（只为 discriminator），
             VerifyResponseV2 对外用组合（meta 信封），两者层次不同，不矛盾
    组合方式：
      class VerifyResponseV2(BaseModel):
          meta: ResponseMeta
          data: Union[VerifyResponse, VerifyResponseMinimal]  # 根据 output_format 分支
  - ResponseMeta：api_version: str / engine_version: str / calc_ms: float
  - 满足 R38（meta.api_version 字段）
