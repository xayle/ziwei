1.1 关键范围决策：Auth/RBAC/数据库
─────────────────────────────────────

  ★ 方案v4.txt 明确写道：
  "不做登录/用户系统/数据库/速率限制/SEO/敏感词扫描/隐私页"

  ★ 但当前代码库已经实现了 auth/RBAC/SQLite/速率限制（半成品）。

  决策: 采用分层策略 ——
    核心层(必做): 命理计算准确性 = 本方案全部精力
    基建层(快修): Auth/DB 中 <5分钟能修的硬伤 → M0 顺手修
    扩展层(延后): RBAC/delegation/权限级联 → 标记为 Phase 2
                  本方案不安排工期，不影响命理核心

  理由: 已有的auth代码不删除(删也要花时间)，
        但不值得投入大量工期去完善一个v4说"不做"的功能。
        只修会导致启动失败/500的硬伤。

  影响: 原91项中约15项auth/RBAC相关 → 8项快修纳入M0，
        7项标记"Phase 2 延后"不占工期。

1.2 架构方向：从v4的16文件 → bazi_engine包
─────────────────────────────────────────────

  方案v4定义了16文件扁平结构。当前代码已偏离此结构。
  本方案采用 services/bazi_engine/ 包结构（20+文件）。

  决策: 正式放弃v4的16文件结构，采用包结构。
  理由: 8维分析引擎 + 生活应用层 无法塞进16文件。
        v4的 bazi.py/verify.py/interpret.py 三文件架构
        适合"最小可用产品"，不适合20Tab完整产品。

  迁移策略:
    - 保留 backends.py / boundary.py / solar_time.py (位置不变)
    - 新建 services/bazi_engine/ 包
    - run.py 中 ~160行内联逻辑 → 抽到 service 层
    - 已有 ganzhi.py/bazi.py/relations.py/interpret.py
      → 标记 deprecated，功能移入 engine 包
    - ENGINE_V2 feature flag 控制新旧切换

  ── 一生运势总论（1个）────────────────────────────────────────────────

  class LifeArcModel(BaseModel):
      """ 一生运势总论（Tab 0总览精简卡片 + Tab 5摘要完整展示） """
      overall_tier:    Literal["局高","局中","局小"]  # 绝大多数命局属"局中"
      early_fortune:   str                            # 幼年/少年（0-20岁）运势（2-3句）
      mid_fortune:     str                            # 青年/中年运势（2-3句）
      late_fortune:    str                            # 晚年运势（2-3句）
      peak_periods:    list[str]                      # 一生最旺1-2步大运，如["甲子大运(36-45岁)"]
      caution_periods: list[str]                      # 需要小心的山穷水尽大运段
      life_motto:      str                            # 此命局适用的一句古籍旗面（古籍引用）
      inference_tags:  list[str]
      interpretation_text: str                        # 100-200字综述
      disclaimer:      str = "仅供学术研究参考"
      optimal_action:  Optional[str] = None  # 当年最佳出手时机（同 LiuNianDetailModel.optimal_action 格式）
      # 规则: 早中晚三段以 dayun 第1-2步 / 3-5步 / 6步后 大致划分
      # overall_tier 四因子评分法:
      #   A: 格局等级  上格+2 / 中格+1 / 下格或无格+0
      #   B: 日主强弱  身强且用财官+1 / 身弱且用印比+1 / 过旺过弱-1
      #   C: 旺运覆盖率  大运步数中吉运≥60%→+2 / 40-59%→+1 / <40%→+0
      #   D: 用神得力  流年/大运与用神五行相合+1
      #   总分≥5→"局高"  3-4→"局中"  ≤2→"局小"
      #   ★ 绝大多数命局总分在3-4分，属"局中"

4.5 错误处理模式（新分析引擎适用）
───────────────────────────────────

  分析引擎(wealth/career/marriage等)的错误处理:
  - 单维度异常 → 该维度返回 null + warnings 追加原因
  - 核心计算异常(四柱/五行/用神) → 整体返回 500
  - 输入校验异常 → 返回 400

  原则: 部分分析失败不应阻断其他维度的正常输出。

4.5.1  扩展错误处理规格（四类补充场景）
──────────────────────────────────────────
  前端错误处理（verify-core.js）:
  ──────────────────────────────────────
    HTTPError 5xx        →  toast("系统繁忙，请稍后重试") + 重试按钮（最多2次自动重试）
    HTTPError 4xx        →  toast(error.detail) + 不自动重试（用户输入问题）
    网络超时(>30秒)      →  skeleton 保持（不清空内容）+ Tab 右上角显示"请求超时"badge
    JSON 解析异常        →  console.error() + 该 Tab 显示"数据解析异常，请刷新重试"
    ★ 任何情况下禁用 alert() / confirm() / prompt() [红线#16]

  缓存层错误（bazi_engine_service.py）:
  ──────────────────────────────────────────
    cache miss           →  正常走引擎计算，透明处理（不报错）
    cache 读取异常       →  静默忽略，继续计算（缓存是性能优化，非核心依赖）
    cache 写入异常       →  静默忽略，正常返回计算结果，下次仍尝试写入
    ★ 缓存层任何异常不得上浮为 5xx/4xx

  SQLite 并发锁（db.py，--workers 4 场景）:
  ──────────────────────────────────────────
    OperationalError("database is locked")
      → 重试 3 次，间隔 100ms（asyncio.sleep(0.1)）
      → 3 次仍失败 → 返回 503 + {"error":"数据库暂时不可用，请重试"}
    ★ 生产启动时必须执行: PRAGMA journal_mode=WAL（在 db.py 连接初始化）
      WAL 模式下并发读写冲突概率极低，但重试机制必须保留

  第三方库异常（cnlunar / sxtwl）:
  ──────────────────────────────────────────
    sxtwl 节气/四柱计算异常  →  500（核心路径，不可降级，关联红线#11和大运逆排）
    cnlunar 时柱映射异常     →  500（核心路径，AssertionError from [P65]断言触发时同样500）
    sxtwl 神煞/格局辅助计算  →  该字段 = null + warnings 追加"神煞计算异常:xxx"（可降级）
    sxtwl 流年辅助计算       →  该年流年部分字段 null + warnings（可降级）
    ★ 核心路径和辅助路径的边界: 四柱/五行/日主/大运 = 核心; 神煞/格局/流年= 辅助

4.11  分析引擎算法规格（7个引擎 · 输入→规则→输出三段式）
─────────────────────────────────────────────────────────
  ★ 以下为各引擎的命理学推导规格，精度对齐 0.22 L级别决策矩阵。
  ★ 所有引擎输出须遵循 §4.6 三层模型（fact/inference/interpretation）。
  ★ 引擎间依赖关系见 §4.12。

  ──────────────────────────────────────────────────────
  A. analysis/wealth.py  财运引擎
  ──────────────────────────────────────────────────────
  输入:
      yongshen:    YongShenModel           # 用神五行（第一/第二用神）
      wuxing:      WuXingScoreModel        # 五行得分（含 scores_weighted）
      shishen:     dict[str, float]        # 十神得分分布（正财/偏财/官杀/印等）
      dayun_list:  list[DaYunItemModel]    # 大运列表（取当前及未来3步评趋势）

  规则:
      1. 财星力量 = 正财得分×1.0 + 偏财得分×0.8（加权后归一化到0-100）
      2. 用神匹配度 = 日主用神五行中含金/水/木/火/土的比例
      3. wealth_score = 财星力量×0.5 + 用神匹配度×0.3 + 月令旺衰系数×0.2
         ★ 身弱财旺反向修正（传统口诀「财多身弱主贫困」）:
           若 strength.score < 30 → wealth_score = wealth_score × (strength.score / 50)
           （日主极弱无力承财，财反为忧，防止高财运误判）
         tier: score≥70='上', score≥40='中', else='下'
      4. 行业映射（取用神五行）:
           金→金融/律政/机械制造/珠宝
           木→教育/医药/农林/食品/出版
           水→贸易/运输/IT/资讯/旅游
           火→传媒/餐饮/能源/娱乐演艺
           土→房地产/保险/建筑/农业
      5. 大运周期: 遍历 dayun_list, 大运天干与日干相生/比和 → trend="上升"
                   相克（大运克日干）→ trend="下降"，其余 → trend="平稳"
         ★ 地支修正（传统口诀「大运地支力量重于天干」）:
           大运地支严冲日支（六冲）→ trend 降一档（上升→平稳，平稳→下降）
           大运地支与用神五行同气或三合 → trend 升一档（下降→平稳，平稳→上升）
           冲日支与升用神同时触发时，以冲日支优先（稳健估计）

  输出 → WealthAnalysisModel（见§4.3.1）
  验证: wealth_score ≠ strength.score（禁止直接照搬日主强弱分）

  ──────────────────────────────────────────────────────
  B. analysis/career.py  事业引擎
  ──────────────────────────────────────────────────────
  输入:
      geju:     GejuModel
      yongshen: YongShenModel
      shishen:  dict[str, float]
      strength: DayMasterStrengthModel

  规则（职业线优先级，取最高项）:
      官星有力（官+杀得分≥60） → 管理/仕途/行政线
      食伤有力（食+伤得分≥60） → 技术/创意/设计/表达线
      财星有力（财得分≥60）    → 商业/销售/投资线
      印星有力（印得分≥60）    → 学术/文职/教育线
      以上均弱                 → 依格局定:
          七杀格→武职/安保/竞技; 食神格→厨艺/手工艺/娱乐
          伤官格→艺术/文创; 正财格→稳定单位/财务
      领导潜力: 官杀有力 AND 日主中和或偏旺 → leadership_potential=True

  输出 → CareerAnalysisModel
  验证: career_directions 不为空列表; suitable_industries 至少2项

  ──────────────────────────────────────────────────────
  C. analysis/marriage.py  婚姻引擎
  ──────────────────────────────────────────────────────
  输入:
      pillars:  dict          # 四柱原始数据（日支=配偶宫）
      shishen:  dict[str, float]
      shensha:  list[ShenshaModel]
      gender:   Literal["male","female"]

  规则:
      1. 配偶宫稳定性（日支分析）:
           日支自刑/被冲/被合化他 → marriage_score 基础-15（不稳定信号）
           日支三合/六合 成局     → marriage_score 基础+10（稳定信号）
      2. 婚姻星力量:
           女命: 正官/七杀（夫星）有力且官杀不混（官≤15 或 杀≤15，两者不同时>15） → 利婚姻+10
                 ★ 官杀混杂预警: 官得分>15 且 杀得分>15 → marriage_note 添加「官杀交加，感情易复杂」
                   传统依据:《渊海子平》「女命官杀交加，克夫再嫁」
           男命: 正财/偏财（妻星）有根且适中（财得分30-70） → 利婚姻+10
      3. 桃花旺衰:
           命中桃花星（子/午/卯/酉，即咸池桃花）在年月日时出现≥2处 → 旺
           出现1处 → 中; 0处 → 弱
           ★ 桃花位置权重说明: 日支/时支桃花 > 月支桃花 > 年支桃花
             （日时近身感情吸引力强；年支属祖上遗传基因，力量最远）
           ★ 大运/流年引动: peach_blossom_note 注明「静桃花（命中）须大运/流年引动方发威」
           ★ 本版仅实现静桃花计算；驿马桃花（寅申巳亥与咸池同柱）留作 v6.0 增强
      4. 配偶五行: 夫星/妻星所对应五行 → partner_wuxing
      5. 婚恋窗口: 大运干支与夫/妻星形成生合关系的区间 → marriage_windows

  输出 → MarriageAnalysisModel
  验证: peach_blossom 在 ["旺","中","弱"] 之一; marriage_windows 非空列表

  ──────────────────────────────────────────────────────
  D. analysis/health.py  健康引擎
  ──────────────────────────────────────────────────────
  输入:
      wuxing:   WuXingScoreModel
      yongshen: YongShenModel
      organ_table: WUXING_TO_ORGAN       # 查找表（M1.04已建于lifestyle/tables.py）

  规则:
      1. 五行偏旺（scores_weighted中该行>40%）→ 对应脏腑"过亢":
           金旺→大肠/肺易积热; 木旺→肝胆易胀; 水旺→肾/膀胱湿寒
           火旺→心脏/小肠易上火; 土旺→脾胃湿热/积滞
      2. 五行偏弱（scores_weighted中该行<10%）→ 对应脏腑"虚损":
           金弱→肺/皮肤虚弱; 木弱→肝血不足; 水弱→肾虚; 火弱→心气虚; 土弱→脾虚泄泻
      3. 风险脏腑列表: 取所有偏旺+偏弱五行对应脏腑，排重
      4. 风险等级: 偏弱≥2个 → 高; 偏弱1个 → 中; 无偏弱 → 低
      5. health_score = 100 - min(五行最大偏差×3, 40)
         （五行标准差每+1个百分点扣3分，最多扣40分）
      6. 养生/运动/饮食: 查 WUXING_TO_LIFESTYLE（取用神五行对应行）

  输出 → HealthAnalysisModel
  验证: risk_organs 不为空（至少1项）; health_score 在0-100

  ──────────────────────────────────────────────────────
  E. analysis/relationship.py  人际引擎
  ──────────────────────────────────────────────────────
  输入:
      shishen:  dict[str, float]
      shensha:  list[ShenshaModel]        # 取天乙贵人/文昌/劫财/羊刃
      gender:   Literal["male","female"]
      pillars:  dict

  规则:
      1. 六亲十神映射（男命）:
           正印=母; 偏印=继母/异母; 正财=妻; 偏财=父/情人
           正官=儿子（阴日主）/女儿; 七杀=儿子（阳日主）; 比肩=兄弟
      2. 六亲十神映射（女命）:

4.12  引擎模块依赖关系图（bazi_engine包内）
─────────────────────────────────────────────
  ★ 只允许向下调用，同层模块互不调用，禁止循环依赖（见§4.12）。

  ────────────────────────────────────────

  ────────────────────────────────────────

    analysis/marriage.py     ← pillars(原始), shishen, shensha
    analysis/health.py       ← wuxing, yongshen, lifestyle/tables

          asyncio.gather(*[wealth, career, marriage, health,
                           relationship, personality, monthly],

          任一引擎异常 → 该维度返回 null + warnings追加（§4.5错误隔离不阻断）

          迁移前提: 所有SQL使用SQLAlchemy ORM（禁止裸SQL字符串）

4.13  新增 API 端点规格
──────────────────────────────────────────
  ── GET /api/v1/glossary  ─────────────────────────────
  描述:  返回命理术语词汇表（供前端 tooltip 使用，见3.05/4.14）
  认证:  无需（公开端点）
  参数:  无强制参数; 可选 ?category=格局|神煞|五行|十神|大运|其他（客户端过滤）

  Response Schema:
    class GlossaryItemModel(BaseModel):
        term:          str                      # 术语，如"七杀"
        pinyin:        str                      # 拼音，如"qī shā"
        definition:    str                      # 简明定义（≤80字）
        category:      Literal["格局","神煞","五行","十神","大运","其他"]
        classic_source: Optional[str] = None   # 来源典籍（如有）

  Response: list[GlossaryItemModel]
  预期数量: ≥50条（3.05任务）
  排序: 按 category 分组，组内按 term 拼音升序
  ★ 不分页（术语数量有限，一次返回全部）
  ★ 数据源: data/glossary.json（静态JSON，随代码库提交）
        服务启动时一次性加载至内存，每次请求不访问DB
        M0阶段创建 data/glossary.json 初始文件（≥50条起）

  ── GET /api/v1/cities  ───────────────────────────────
  描述:  返回城市经纬度列表（供城市选择器使用，见4.24）
  认证:  无需（公开端点）
  参数:  无

  Response Schema:
    class CityModel(BaseModel):
        name:      str     # 城市名，如"北京"
        province:  str     # 省份，如"北京市"
        lng:       float   # 经度，精确到小数点后2位，范围[73.0, 135.5]
        lat:       float   # 纬度
        city_type: Literal["直辖市","省会","计划单列市"]

  Response: list[CityModel]
  数量: 恰好36城（4直辖+27省会+5计划单列）
  排序: 按 city_type 分组，组内按拼音升序
  ★ 数据源: data/cities.json（静态JSON，36行对象数组，随代码库提交）
        服务启动时一次性加载至内存，不访问DB
        注: 附录B.6是WUXING_TO_NUMBER数字表，与城市无关，勿混淆

第八章  风险管理与回滚策略
═══════════════════════════════════════════════════════════════════════

8.1 版本控制策略
────────────────

  分支模型: 简化 Git Flow
    main          — 稳定发布分支
    dev           — 开发主分支
    feat/M0-fixes — 每个里程碑一个 feature branch
    feat/M1-engine
    feat/M2-analysis
    ...

  规则:
  - 每个里程碑完成 → PR 合入 dev → 验收门全通过 → merge
  - 每个 PR 包含: 代码 + 测试 + 验收截图/日志
  - dev 上跑 pytest 通过后才允许 merge 到 main
  - 每个里程碑完成后打 tag: v7.0-M0, v7.0-M1, ...

8.2 里程碑失败回滚
──────────────────

  如果 MN 验收门未通过:
  1. 不合入 dev，继续在 feat/MN-xxx 上修复
  2. 如超过预算2天仍未通过 → 暂停，评估是否简化范围
  3. 后续里程碑自动后移(不压缩)

  如果 M1(最关键) 延期:
  - M1 最多可延3天(→Day 19)
  - M2 起始相应后移，总工期上限 45天
  - 超过45天 → 砍掉 M2.5(生活应用层) 减3天

8.3 ENGINE_V2 Feature Flag
──────────────────────────

  切换粒度: 整个引擎包
  开关: os.getenv("ENGINE_V2", "false") == "true"
  - False → 走旧路径(当前 bazi_full_service.py)
  - True → 走新路径(services/bazi_engine/)
  M1 完成后切为 True，旧代码保留但标 deprecated

8.4 数据兼容
────────────

  snapshot 表存储的旧 JSON:
  - 新增 schema_version 字段(default="legacy")
  - 旧数据只读展示(不走新引擎重算)
  - 新数据标记 schema_version="v7.0"

8.5 工期风险缓冲
────────────────

  | 风险            | 概率 | 影响 | 缓冲策略                |
  |─────────────────|──────|──────|────────────────────────|

  R3  [M0·0.05] SECRET_KEY vs JWT_SECRET_KEY 修正方向不明确
      现状: auth_service.py 读 os.getenv("SECRET_KEY", ...)
            app/config.py   读 os.getenv("JWT_SECRET_KEY", ...)
            → 两个不同环境变量名，F5任务说"统一"但没说往哪统一
      影响: 开发者可能把 app/config.py 改成读 SECRET_KEY，
            也可能把 auth_service.py 改成读 JWT_SECRET_KEY，
            结果两个人改了两个不同的位置
      修正: 明确决策 ——
            保留 os.getenv("SECRET_KEY")，
            app/config.py 的字段名改为 secret_key，
            删除 JWT_SECRET_KEY 变量名；
            .env.example 只保留 SECRET_KEY 一个 key

  R4  [M0·0.22] Single 模式下 L 级别判定语义错误
      现状: 计划0.22修复代码:
              if len(diff_fields) == 0: level = "L0"
            但 mode=single 时 diff_fields 永远为 []，
            这会让单库模式也得到 L0（最高质量标记）
            而当前代码 single→L1（更诚实）
      影响: 前端把 L0 展示为"双库校验通过"，single 模式拿到
            L0 是错误的正向信号，用户被误导
      修正: 0.22 决策矩阵补充 mode=single 分支:
              if mode == "single":           level = "L1"  # 无对比数据
              elif len(diff_fields) == 0:    level = "L0"
              elif set(diff_fields) <= {"hour"}: level = "L2"
              elif set(diff_fields) <= {"hour","month"}: level = "L2"
              else:                          level = "L3"
            红线#35 同步补：L0 必须 mode=dual

  R5  [M0·0.37 完成后才能做] gender 字段添加顺序与 Schema 重构顺序冲突
      现状: 0.37 Schema 合并在 M0 Day 4

            bazi_full_service.py 从根 schemas 导入 BaziFullRequest
            → 两份独立代码，只改一处会造成"missing field gender"运行时错误
      影响: M1 任务 1.13 执行时若 0.37 未完成，字段只加了一半
      修正: 1.13 明确前置条件 "须在0.37合并完成后执行"；

  R6  [M1·1.02后补充切换步骤] solar_time_v2.py 创建后无人调用
